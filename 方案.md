## 五、全自动部署流水线（后台核心，全 gcloud 命令）

### 工作机制

1. **Cloud Function 监听**：用户上传文件到 `gs://customer/projects/` 自动触发
2. **下载文件**：从 customer 桶拉取用户镜像 + YAML
3. **解析配置**：读取项目 ID、二级域名、环境变量
4. **全流程部署**：复用你原有脚本逻辑，100% gcloud 命令
5. **域名配置**：自动绑定二级域名 + SSL+DNS
6. **结果返回**：生成访问地址

### 完整自动化部署脚本（`auto-deploy-customer.sh`）

bash

运行

`#!/bin/bash
# 后台全自动部署脚本（Cloud Function/服务器定时执行）
set -euo pipefail

# ====================== 固定全局配置（你的GCP参数） ======================
REGION="asia-east1"
GCP_PROJECT="project-a8f8e3c3-9724-4306-9bb"
GCS_BUCKET="customer"
ROOT_DOMAIN="aihubflux.com"
AR_REPO="my-app-repo"
VPC_CONNECTOR="nexus-connector-v2"
REGISTRY="${REGION}-docker.pkg.dev"
# ==================================================================

# 1. 遍历customer桶中所有用户项目
echo "🔍 扫描用户项目..."
for USER_PROJECT in $(gsutil ls gs://${GCS_BUCKET}/projects/ | sed 's/gs:\/\/customer\/projects\///g' | sed 's/\///g'); do
  echo "============================================="
  echo " 开始部署用户项目：${USER_PROJECT}"
  echo "============================================="

  # 本地临时目录
  TMP="./tmp-${USER_PROJECT}"
  mkdir -p ${TMP}

  # 2. 从customer桶下载用户所有文件
  gsutil cp gs://${GCS_BUCKET}/projects/${USER_PROJECT}/* ${TMP}/
  CONFIG="${TMP}/user-config.yaml"
  FRONTEND_TAR=$(ls ${TMP}/*frontend*.tar | head -n1)
  BACKEND_TAR=$(ls ${TMP}/*backend*.tar | head -n1)

  # 3. 解析用户YAML配置
  USER_SUBDOMAIN=$(yq e '.user.subdomain' ${CONFIG})
  FRONTEND_IMG=$(yq e '.image.frontend_name' ${CONFIG})
  BACKEND_IMG=$(yq e '.image.backend_name' ${CONFIG})
  TAG=$(yq e '.image.tag' ${CONFIG})
  FULL_DOMAIN="${USER_SUBDOMAIN}.${ROOT_DOMAIN}"

  # 服务名称（用户项目唯一标识）
  FRONTEND_SERVICE="${USER_PROJECT}-frontend"
  BACKEND_SERVICE="${USER_PROJECT}-backend"

  # 4. Docker认证
  gcloud auth configure-docker ${REGISTRY} --quiet

  # ====================== 部署后端服务 ======================
  echo "🚀 部署后端服务：${BACKEND_SERVICE}"
  docker load -i ${BACKEND_TAR}
  BACKEND_FULL_IMAGE="${REGISTRY}/${GCP_PROJECT}/${AR_REPO}/${BACKEND_IMG}:${TAG}"
  docker tag ${BACKEND_IMG}:amd64 ${BACKEND_FULL_IMAGE}
  docker push ${BACKEND_FULL_IMAGE}

  # 生成环境变量
  yq e '.env | to_entries | map("\(.key): \"\(.value)\"") | .[]' ${CONFIG} > ${TMP}/env.yaml

  # 部署Cloud Run
  gcloud run deploy ${BACKEND_SERVICE} \
    --image=${BACKEND_FULL_IMAGE} \
    --platform=managed --region=${REGION} \
    --allow-unauthenticated --port=3000 \
    --cpu=2 --memory=4Gi --min-instances=1 \
    --vpc-connector=${VPC_CONNECTOR} --vpc-egress=private-ranges-only \
    --env-vars-file=${TMP}/env.yaml --quiet

  # ====================== 部署前端服务 ======================
  echo "🚀 部署前端服务：${FRONTEND_SERVICE}"
  docker load -i ${FRONTEND_TAR}
  FRONTEND_FULL_IMAGE="${REGISTRY}/${GCP_PROJECT}/${AR_REPO}/${FRONTEND_IMG}:${TAG}"
  docker tag ${FRONTEND_IMG}:amd64 ${FRONTEND_FULL_IMAGE}
  docker push ${FRONTEND_FULL_IMAGE}

  gcloud run deploy ${FRONTEND_SERVICE} \
    --image=${FRONTEND_FULL_IMAGE} \
    --platform=managed --region=${REGION} \
    --allow-unauthenticated --port=80 \
    --cpu=1 --memory=2Gi --min-instances=1 \
    --quiet

  # ====================== 自动配置二级域名（核心） ======================
  echo "🌐 配置二级域名：${FULL_DOMAIN}"
  # 绑定前端服务到二级域名（自动签发SSL）
  gcloud run domain-mappings create \
    --service=${FRONTEND_SERVICE} \
    --region=${REGION} \
    --domain=${FULL_DOMAIN} \
    --quiet

  # 自动配置DNS解析
  RUN_TARGET=$(gcloud run domain-mappings describe --domain=${FULL_DOMAIN} --region=${REGION} --format='value(resourceRecords[0].rrdata)')
  gcloud dns record-sets transaction start --zone=${DNS_ZONE}
  gcloud dns record-sets transaction add ${RUN_TARGET} \
    --name=${FULL_DOMAIN}. --ttl=300 --type=A --zone=${DNS_ZONE}
  gcloud dns record-sets transaction execute --zone=${DNS_ZONE}

  # 5. 清理临时文件
  rm -rf ${TMP}

  echo "✅ 用户项目 ${USER_PROJECT} 部署完成！"
  echo "🌍 访问地址：https://${FULL_DOMAIN}"
done`

## 七、关键注意事项（生产必看）

### 1. 权限安全（最重要）

- **用户无任何 GCP 权限**：仅通过面板 / 工具上传文件，无法访问 GCP 控制台
- `customer`桶权限：仅后台服务账号可读写，用户无法查看他人项目
- 镜像仓库：所有用户镜像统一存储，后台自动隔离

### 2. 二级域名管理

- 自动校验**二级域名唯一性**，禁止重复
- 二级域名规则：小写字母 + 数字 + 短横线，长度 3-20 位
- 自动签发**免费 Google 托管 SSL 证书**，HTTPS 默认启用

### 3. 资源隔离

- 每个用户项目对应**独立 Cloud Run 服务**
- 服务命名规则：`{用户项目ID}-frontend/backend`，绝对不冲突
- 复用 VPC / 镜像仓库，降低成本，资源隔离

### 4. 稳定性 & 故障处理

- 脚本添加`set -euo pipefail`，出错自动终止，避免脏部署
- 上传失败：面板提示用户重新上传
- 域名生效延迟：DNS 解析 1-5 分钟，SSL 证书 30 秒 - 2 分钟
- 日志：所有部署日志存储在 `gs://customer/logs/`，可排查问题

### 5. 兼容前后端

- 前端：默认端口 80，静态服务 /nginx 镜像
- 后端：默认端口 3000，复用你原有环境变量配置
- YAML 自动适配前后端环境变量，无需修改脚本

### 6. 成本优化

- 所有服务`min-instances=1`，空闲不收费
- 统一存储 / 镜像仓库，无额外成本
- Cloud Run 按需扩容，自动缩容
